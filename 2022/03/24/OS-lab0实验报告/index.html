Thinking 0.3

1.深夜，小明在做操作系统实验。困意一阵阵袭来，小明睡倒在了键盘上。等到小明早上醒来的时候，他惊恐地发现，他把一个重要的代码文件printf.c删除掉了。苦恼的小明向你求助，你该怎样帮他把代码文件恢复呢？

使用git checkout -- printf.c可以找回最近一次add后在暂存区的文件。

2.正在小明苦恼的时候，小红主动请缨帮小明解决问题。小红很爽快地在键盘上敲下了git rm printf.c，这下事情更复杂了，现在你又该如何处理才能弥补小红的过错呢？

使用git checkout HEAD printf.c可以找回最近一次commit后在master分支中的的文件。

或者使用git reset HEAD printf.c; git checkout -- printf.c 先恢复暂存区，再从暂存区恢复工作区文件。

3.处理完代码文件，你正打算去找小明说他的文件已经恢复了，但突然发现小明的仓库里有一个叫 Tucao.txt，你好奇地打开一看，发现是吐槽操作系统实验的，且该文件已经被添加到暂存区了，面对这样的情况，你该如何设置才能使Tucao.txt在不从工作区删除的情况下不会被 git commit 指令提交到版本库？

使用git rm --cache Tucao.txt将缓存区中的文件清除（进而不会被commit到版本库）而又不影响工作区中的文件。

Thinking 0.5

思考下面四个描述，你觉得哪些正确，哪些错误，请给出你参考的资料或实验证据。

1.克隆时所有分支均被克隆，但只有 HEAD 指向的分支被检出。

远程仓库的所有分支会作为本地仓库的远程跟踪分支(remote-tracking branches)被创建，但本地分支只会创建一个，为远程仓库当前HEAD指向的分支。并且会检出到这个本地分支。

Clones a repository into a newly created directory, creates remote-tracking branches for each branch in the cloned repository (visible using git branch --remotes), and creates and checks out an initial branch that is forked from the cloned repository’s currently active branch.

——来自git官网

2.克隆出的工作区中执行 git log、git status、git checkout、git commit 等操作不会去访问远程版本库。

正确。

由于git是分布式版本控制软件，git clone会将远程仓库的所有数据完整的克隆下来，因此git log git status不需要访问远程库即可查询到相应的信息。对git checkout，如果检出分支不存在且存在名字与之匹配的唯一远程追踪分支，就会自动创建本地分支。这一过程中所有数据均存在于本地，故不需要访问远程库。对git commit其提交过程是对于本地仓库的，故也不需要访问远程库。

当克隆一个仓库时，它通常会自动地创建一个跟踪 origin/master 的 master 分支。 然而，如果你愿意的话可以设置其他的跟踪分支，或是一个在其他远程仓库上的跟踪分支，又或者不跟踪 master 分支。 最简单的实例就是像之前看到的那样，运行 git checkout -b <branch> <remote>/<branch>。 这是一个十分常用的操作所以 Git 提供了 --track 快捷方式：

由于这个操作太常用了，该捷径本身还有一个捷径。 如果你尝试检出的分支 (a) 不存在且 (b) 刚好只有一个名字与之匹配的远程分支，那么 Git 就会为你创建一个跟踪分支：

——Pro Git

实验难点

Exercise 0.4

在Makefile文件的理解和编写上掌握的不够。最开始打算通过Makefile的嵌套调用完成程序的编译工作，但是在Makefile内部切换目录并执行次级Makefile时会出现错误，当时没有找到解决方法故作罢，改成仅用外层Makefile执行全部操作。后来通过和同学交流才知道，在Makefile文件中，cd命令换行后，后续命令的执行目录仍然为当前Makefile文件的执行目录，而非cd切换后的目录。为了解决这一问题，需要在cd命令后加&& make。

此外，Makefile十分丰富的语法细节也不太好在短时间内掌握，写出一个可读性和可维护性比较强的Makefile不太容易。
